# Spec-Kit + Superpowers 完整开发流程文档

## 一、文档说明

本文档适用于 Superpowers v5.x 版本（无斜杠命令，自然语言触发技能），详细梳理 Spec-Kit 与 Superpowers 搭配的全流程，聚焦「全项目规范管控+高质量开发」，适用于中大型项目、跨团队协作、全流程可追溯的场景。

核心定位：Spec-Kit 定义「做什么、怎么约束」，Superpowers 执行「怎么做、怎么落地」，全程无需手动输入斜杠命令，AI 自动触发对应技能。

## 二、核心角色分工

- **Spec-Kit**：规范制定者（甲方），负责定义「做什么」和「怎么约束」
  - 制定项目宪法、全局规则、功能规范
  - 锁死需求边界，冻结后禁止自由修改
  - 任何变更必须先改 Spec，再同步代码

- **Superpowers**：严格执行引擎（乙方），负责执行「怎么做」和「怎么落地」
  - 读取 Spec-Kit 规范，进入严格执行模式
  - 基于规范以 TDD 模式完成开发、测试、审查
  - 全程不自由发挥、不偏离规范，确保 100% 合规

## 三、完整开发流程（6大阶段，可直接落地）

### 阶段1：项目初始化 & 全局规则锁定（Spec-Kit 主导）

#### 目标

建立项目级「宪法」，统一技术栈、架构、编码规范，从源头杜绝开发混乱，明确全局约束。

#### 操作步骤

**1. 初始化 Spec-Kit 项目**

```
用 Spec-Kit 初始化新项目，生成标准目录结构：
- 根目录 CONSTITUTION.md（项目宪法）
- specs/ 目录（存放所有功能规范）
- tasks.md（开发任务清单）
```

**2. 定义全局约束（写入 CONSTITUTION.md）**

```
在 CONSTITUTION.md 中明确：
1. 技术栈：Node.js + Express + JWT + MongoDB
2. 架构约束：三层架构（Controller/Service/Repository）
3. 编码规范：ESLint 严格、禁止 TODO、必须 TDD
4. 安全要求：密码 bcrypt 加密、参数校验、日志审计
5. 协作规则：所有变更必须先改 Spec，再改代码
```

**3. 确认全局规则生效**

```
校验 CONSTITUTION.md 格式合规，确认全局规则冻结，后续开发必须严格遵守
```

---

### 阶段2：功能需求定义 & 规范固化（Spec-Kit 主导）

#### 目标

把模糊需求转化为结构化、可执行、可验证的规范，锁死边界。

#### 操作步骤

**1. 定义单个功能的完整规范**

```
用 Spec-Kit 定义【XXX 模块】的完整规范（以用户登录为例，可替换为实际模块）：
- 接口：POST /api/xxx（根据实际模块定义）
- 请求字段：xxx（必填，字符串，根据实际字段定义）
- 响应结构：200={...}、400={error:xxx}、401={error:xxx}（根据实际响应定义）
- 业务规则：根据实际业务逻辑定义
- 验收标准：100% 符合接口规范、测试覆盖率 100%
```

**2. 生成规范文件**

```
AI 自动生成 specs/auth/login.spec.md（或 YAML 格式），包含所有字段、规则、验收标准
```

**3. 规范校验 & 冻结**

```
用 Spec-Kit 校验所有规范的合法性，确认无冲突、无遗漏，执行规范冻结
✅ 冻结后：所有开发必须严格遵循，禁止自由修改结构 / 字段 / 逻辑
```

---

### 阶段3：开发计划生成（Spec-Kit + Superpowers 交接）

#### 目标

从规范拆解出可执行的开发任务，完成「需求→开发」的无缝衔接。

#### 操作步骤

**1. Spec-Kit 拆解任务**

```
用 Spec-Kit 基于当前规范，生成详细的开发任务清单 tasks.md（如：从 XXX 模块规范拆解出对应的任务）
```

**2. Superpowers 接手规范 √**

```
用 Superpowers from-spec 模式，读取当前 Spec-Kit 所有规范，进入严格执行模式
要求：禁止自由设计、禁止修改规范、100% 按 Spec 开发
```

**3. 生成开发执行计划**

```
AI 自动生成包含「测试→编码→审查」的完整开发计划，对齐 Spec 约束
```

---

### 阶段4：TDD 模式高质量开发（Superpowers 主导）

#### 目标

用强制 TDD 流程保证代码质量，完全符合规范。

#### 操作步骤

**1. 开启 TDD 强制模式 √**

```
用 Superpowers TDD 流程开发 当前模块：
1. 先基于 Spec 生成完整测试用例（正常/异常/边界场景）
2. 测试用例必须覆盖所有 Spec 定义的字段、状态码、规则
3. 测试不通过，禁止编写实现代码
```

**2. 分层实现代码**

```
AI 按架构约束，自动生成：
- 接口层（Controller）
- 业务层（Service）
- 数据层（Repository）
- 参数校验中间件
- 错误处理逻辑
```

**3. 自动代码审查 √**

```
用 Superpowers 审查代码，检查：
- 是否完全符合 Spec-Kit 规范
- 是否遵循架构约束
- 是否存在安全隐患
- 是否符合编码规范
```

**4. 修复问题**

```
AI 自动修复审查中发现的所有偏离规范的问题，确保代码质量达标
```

---

### 阶段5：全链路合规审计（Superpowers 主导）

#### 目标

最终校验代码与规范 1:1 对齐，杜绝偏离。

#### 操作步骤

**1. 执行全量审计 √**

```
用 Superpowers 审计当前项目所有代码：
1. 检查接口路径、字段、返回码是否与 Spec 完全一致
2. 检查业务逻辑是否符合 Spec 定义的规则
3. 检查测试用例是否覆盖所有 Spec 场景
4. 检查架构是否符合 CONSTITUTION.md 约束
```

**2. 修复审计问题**

```
AI 自动修复所有偏离规范的问题，确保 100% 合规
```

**3. 生成审计报告**

```
输出合规性报告，作为上线依据
```

---

### 阶段6：需求变更管理（Spec-Kit + Superpowers 协同）

#### 目标

规范变更流程，避免代码混乱。

#### 操作步骤

**1. 变更必须先改 Spec**

```
用 Spec-Kit 修改【XXX 模块】规范（如：新增字段、修改业务规则）
```

**2. 重新校验 & 冻结规范**

```
用 Spec-Kit 校验修改后的规范，确认无冲突，重新冻结
```

**3. Superpowers 同步更新**

```
用 Superpowers from-spec 重新读取规范，同步更新代码、测试用例，确保对齐新规范
```

---

## 四、注意事项

- 全程无需输入 /superpowers 斜杠命令，通过自然语言触发 Superpowers 技能即可。

- Spec 规范冻结后，禁止随意修改；若需变更，必须先修改 Spec，再同步代码。

- TDD 模式为强制要求，必须先写测试用例，再编写实现代码，确保测试覆盖率 100%。

- 每次开发完成后，必须执行 Superpowers 审计，确保代码 100% 合规。
