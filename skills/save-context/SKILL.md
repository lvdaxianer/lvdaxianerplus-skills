# 保存重点到 Context 文件

## Overview

当用户在项目中输入包含"重点："前缀的内容时，自动将重点保存到项目根目录下的 `.ai/` 目录中的 context 文件。

## When to Use

- 用户输入 "重点：xxx" 时
- 用户输入 "重点-模块名：xxx" 时

## 支持的格式

| 输入格式 | 保存位置 | 示例 |
|---------|---------|------|
| `重点：xxx` | `.ai/context.md` | 重点：这是通用重点 |
| `重点-模块名：xxx` | `.ai/模块名-context.md` + `.ai/context.md` | 重点-editor：编辑器相关重点 |

## Core Actions

### 1. 解析用户输入

检测输入是否匹配以下模式：

- **通用格式**: `重点：<内容>` → 只保存到 `.ai/context.md`
- **模块格式**: `重点-<模块名>：<内容>` → 同时保存到 `.ai/<模块名>-context.md` 和 `.ai/context.md`

**正则表达式**:
```
^重点(?:-([^：]+))?：(.+)$
```

- 匹配组1: 模块名（可选）
- 匹配组2: 重点内容

### 2. 确定目标文件

1. 获取当前项目的根目录路径
2. 检查 `.ai` 目录是否存在，不存在则创建
3. 根据是否有模块名确定目标文件：
   - 无模块: 只更新 `context.md`
   - 有模块: 同时更新 `<模块名>-context.md` 和 `context.md`

### 3. 读取或创建文件

**如果文件不存在**，创建新文件并写入标题：

```markdown
# {模块名}重点

## {日期}

- {内容}
```

**如果文件已存在**，检查是否需要新的日期分组：

1. 读取文件内容
2. 查找最后一个 `##` 标题（日期）
3. 如果日期是今天 → 追加到该日期下
4. 如果日期不是今天 → 创建新的日期分组

### 4. 追加内容

使用以下格式追加重点：

**模块文件 (如 editor-context.md):**
```markdown
## 2026-03-13

- 编辑器相关的重点
```

**通用文件 (context.md) - 有模块时标注来源:**
```markdown
## 2026-03-13

- [Editor] 编辑器相关的重点
- 这是一个通用重点
```

### 5. 写入文件

将更新后的内容写回文件，并告知用户保存结果。

## 双文件更新逻辑

当用户输入 `重点-editor：xxx` 时：

1. 解析输入，提取模块名 "editor" 和内容 "xxx"
2. 更新 `.ai/editor-context.md`:
   - 标题: `# Editor 重点`
   - 内容: `- 编辑器相关的重点`
3. 同时更新 `.ai/context.md`:
   - 标题: `# 项目重点`
   - 内容: `- [Editor] 编辑器相关的重点`（带模块标签）

## 文件格式示例

### 通用 context - .ai/context.md

```markdown
# 项目重点

## 2026-03-13

- 这是一个测试重点
- [Editor] 编辑器相关的重点

## 2026-03-14

- 新增的重点
```

### 模块 context - .ai/editor-context.md

```markdown
# Editor 重点

## 2026-03-13

- 编辑器相关的重点
```

## 实现步骤

1. 使用正则表达式解析用户输入，提取模块名和内容
2. 获取当前工作目录作为项目根目录
3. 构建 `.ai` 目录的完整路径
4. 确保 `.ai` 目录存在
5. 根据模块名确定目标文件名
6. 读取目标文件（如存在）或创建新文件结构
7. 将重点追加到当天日期分组下
   - 有模块时: 同时更新两个文件
   - 无模块时: 只更新 context.md
8. 写入文件并确认保存成功

## 错误处理

- 如果无法创建目录，提示用户检查权限
- 如果无法写入文件，提示用户检查文件权限
- 如果正则不匹配，忽略该输入（不处理）
- 如果用户输入 `重点：` 后面没有内容，提示用户输入有效内容

## 额外处理规则

### 1. 模块名验证

模块名需要做安全处理：
- 移除或替换不允许的文件名字符（如 `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`）
- 限制模块名长度（最大 50 字符）
- 模块名首字母大写（如 `editor` → `Editor`）

### 2. 多行内容处理

如果用户输入包含换行：
- 将换行转换为空格连接成单行
- 或者在Markdown中使用 `>` 引用格式保留换行

### 3. 标题生成规则

- 无模块: `# 项目重点`
- 有模块: `# {模块名} 重点`（首字母大写）

### 4. 工作目录获取

使用 Claude Code 的工具（如 `Bash pwd` 或环境变量）获取当前工作目录作为项目根目录。

### 5. 来源标注

在 `context.md` 中记录有模块的重点时，使用 `[模块名]` 格式标注来源：
- `[Editor] 编辑器相关的重点`
- `[API] 接口相关的重点`

## 最佳实践案例

### 场景一：记录通用重点

用户在开发过程中发现了一个重要的设计原则，需要记录下来：

```
用户输入: 重点：所有接口必须返回统一的响应格式，包含 code、message、data 字段
```

**生成文件 .ai/context.md:**
```markdown
# 项目重点

## 2026-03-13

- 所有接口必须返回统一的响应格式，包含 code、message、data 字段
```

### 场景二：记录模块特定重点

用户在开发编辑器功能时，发现了一个重要的实现细节，需要同时记录到通用文件和编辑器专用文件：

```
用户输入: 重点-editor：编辑器文本必须使用 Virtual DOM 进行高效渲染
```

**生成文件 .ai/editor-context.md:**
```markdown
# Editor 重点

## 2026-03-13

- 编辑器文本必须使用 Virtual DOM 进行高效渲染
```

**生成文件 .ai/context.md:**
```markdown
# 项目重点

## 2026-03-13

- [Editor] 编辑器文本必须使用 Virtual DOM 进行高效渲染
- 所有接口必须返回统一的响应格式，包含 code、message、data 字段
```

### 场景三：记录多个模块重点

用户继续记录其他模块的重点：

```
用户输入: 重点-api：RESTful API 必须使用名词复数形式，如 /users 而非 /getUser
```

**生成文件 .ai/api-context.md:**
```markdown
# API 重点

## 2026-03-13

- RESTful API 必须使用名词复数形式，如 /users 而非 /getUser
```

**生成文件 .ai/context.md:**
```markdown
# 项目重点

## 2026-03-13

- [Editor] 编辑器文本必须使用 Virtual DOM 进行高效渲染
- 所有接口必须返回统一的响应格式，包含 code、message、data 字段
- [API] RESTful API 必须使用名词复数形式，如 /users 而非 /getUser
```

### 场景四：常用模块名推荐

| 模块名 | 适用场景 |
|--------|----------|
| `editor` | 编辑器、富文本相关 |
| `api` | 接口、API 设计相关 |
| `db` | 数据库、ORM 相关 |
| `auth` | 认证、授权相关 |
| `config` | 配置、环境相关 |
| `deploy` | 部署、CI/CD 相关 |
| `perf` | 性能优化相关 |
| `security` | 安全相关 |
